## 1. Core Features

OpenDroid leverages a hybrid execution flow to run tasks reliably:
*   **Direct Command Aliasing**: Common phrases (e.g., "toggle flashlight", "settings") are intercepted and resolved locally without needing an LLM call.
*   **Structured Planning**: Complex queries (e.g., "take a screenshot and message it to Dad") are planned by a local/remote LLM into sequential steps.
*   **Self-Contained Actions**: Critical intents (like Calls and SMS) do not require opening their respective apps first. The agent handles deep links or fallback intents natively to speed up execution.
*   **Action Auto-Mapper**: Automatically corrects minor typo variations or hallucinated actions generated by the planner before execution.

## 2. Communication Commands

OpenDroid features robust, fallback-oriented communication controls:

| Action Name | Parameters | Behavior |
| :--- | :--- | :--- |
| **`MAKE_CALL`** | `contact` (or `number` / `phone`) | Resolves the contact using the fuzzy resolver. Starts direct background calling if the `CALL_PHONE` permission is granted, otherwise opens the native system dialer with the number pre-filled. |
| **`SEND_SMS`** | `contact`, `message` (or `text` / `body`) | Automatically attempts direct SMS sending. If cellular services are unavailable (e.g., on tablets or emulator environments), it falls back to launching the default SMS composer app with pre-filled details. |
| **`SEND_WHATSAPP`** | `contact`, `message` | Opens WhatsApp directly using the contact's phone number. Pre-fills the message and automatically clicks the "Send" button using the Accessibility service. |
| **`SEND_EMAIL`** | `to`, `subject`, `body` | Drafts and dispatches emails using default system mail clients. |

## 3. System & Settings Actions

OpenDroid can manage system hardware and system states locally:

*   **`TOGGLE_FLASHLIGHT`**: Correctly toggles the camera torch between `ON` and `OFF` by listening to camera hardware state callbacks (`isFlashlightOn`).
*   **`TOGGLE_WIFI`**: Switches the Wi-Fi state.
*   **`TOGGLE_BLUETOOTH`**: Switches the Bluetooth state.
*   **`TOGGLE_DND`**: Toggles Do Not Disturb (DND) mode (`on` or `off`).
*   **`TOGGLE_HOTSPOT`**: Enables or disables the mobile hotspot.
*   **`OPEN_APP`**: Launches any installed application by its friendly name (e.g. "WhatsApp", "Google Maps").

## 4. Advanced Control & File System Actions

Advanced controls allow OpenDroid to interact directly with the screen layout and manage local storage files.

### Layout & Screen Automation
*   **`CLICK_TEXT`**: Scans the active screen window via accessibility nodes and clicks any element containing the specified text.
*   **`CLICK_ID`**: Clicks a view component by its exact resource ID.
*   **`TYPE_TEXT`**: Searches for an editable input containing a placeholder text and types custom content into it.
*   **`TYPE_ID`**: Types custom content into a field matching a specific resource ID.
*   **`SCROLL`**: Scrolls `forward` or `backward` through scrollable screen containers.
*   **`CLICK_COORDINATES`**: Emulates a touch event at a specific `(x, y)` coordinate.
*   **`GET_SCREEN_TEXT`**: Scrapes all visible text contents from the current screen and returns it.

### Background Tools
*   **`TAKE_PHOTO_BACKGROUND`**: Captures a still image from the back camera lens silently in the background. If camera access is blocked, it seamlessly falls back to opening the system Camera app.
*   **`GET_SYSTEM_INFO`**: Retrieves hardware specifications (Battery level, Storage size, Free Memory size, Network connection type, OS version, Manufacturer, and Model).

### File Management
*   **`LIST_FILES`**: Lists all directories and files at a given system path.
*   **`READ_FILE`**: Reads text files from local storage (up to 100KB size limit).
*   **`WRITE_FILE`**: Creates a file and writes content to it.
*   **`DELETE_FILE`**: Deletes a file or directory recursively.
*   **`CREATE_DIRECTORY`**: Creates folders and parent sub-folders.
*   **`COPY_FILE` / `MOVE_FILE`**: Copies or moves files across directories.
*   **`ZIP_FILES` / `UNZIP_FILE`**: Compresses or extracts `.zip` files.

## 5. Macros: Creation & Automation

OpenDroid supports **Macros**—user-defined or system-defined sequences of actions executed sequentially or on a schedule.

### Macro Actions

1.  **`CREATE_MACRO`**
    *   **Parameters**:
        *   `name`: Unique identifier for the macro (e.g., `"morning_routine"`).
        *   `steps`: A JSON array of actions with their arguments.
    *   **JSON Format**:
        ```json
        [
          { "action": "TOGGLE_FLASHLIGHT", "params": { "on": "true" } },
          { "action": "SET_RINGER_MODE", "params": { "mode": "normal" } }
        ]
        ```

2.  **`RUN_MACRO`**
    *   **Parameters**:
        *   `macroName`: Name of the macro to trigger.
    *   **Behavior**: Retrieves steps from the database and executes them.

3.  **`SCHEDULE_MACRO`**
    *   **Parameters**:
        *   `macroName`: Name of the macro to schedule.
        *   `cronExpression`: Standard cron syntax string.
    *   **Behavior**: Attaches a cron schedule trigger (e.g. `cron:0 8 * * *` for daily at 8:00 AM) to automatically fire the macro steps.

## 6. Vision Engine: Screen Analysis

OpenDroid integrates a multimodal **Vision Engine** for visual reasoning.

*   **Action**: `ANALYZE_SCREENSHOT` (or voice trigger: *"Take a screenshot and analyze it"*)
*   **Dual-Tier Execution**:
    1.  **Tier 1 (Image Analysis)**: The engine captures the active screen buffer via `AccessibilityService.takeScreenshot()` (supported on Android 11/API 30+), encodes it into base64, and dispatches it to a multimodal LLM for visual analysis.
    2.  **Tier 2 (Text-Only Fallback)**: On older Android versions (API < 30) or if a screenshot is temporarily blocked, the engine automatically scrapes all visible accessibility nodes (`getScreenText()`) and feeds the raw layout hierarchy to the LLM.
*   **Use Cases**:
    *   Reading active chat windows.
    *   Describing open settings menus.
    *   Translating dynamic screen contents.

## 7. Fuzzy Contact Resolver

The **ContactResolver** acts as a bridge between the agent and your phone book:
1.  **Exact Match**: Tries to match the search string directly with a contact name.
2.  **Partial/Fuzzy Match**: Tries to identify similar contact names (e.g. matching `"dad"` or `"dady"` to `"Dad"`).
3.  **Error Handling**: If no contact is resolved and the input is not a direct phone number, it stops execution and warns: `Contact '<name>' not found in your contacts` rather than launching empty intents.

## 8. Troubleshooting & Permission Guides

To run all features, ensure the following settings are enabled:

### ⚙️ Required System Permissions
1.  **Accessibility Service**:
    *   Required for screenshot analysis, screen text scraping, clicks (`CLICK_TEXT`, `CLICK_ID`), and typing.
    *   Go to **Settings > Accessibility > Installed Services > OpenDroid** and toggle to **ON**.
2.  **Files & Storage**:
    *   Required for file read/write operations.
    *   Ensure "All Files Access" is toggled for OpenDroid.
3.  **Call & SMS**:
    *   Ensure permissions are allowed. If denied, the app will run the interactive fallback intents (Dialer / SMS composer).
4.  **DND Policy**:
    *   Required for `SET_RINGER_MODE`. The app will redirect you to permission settings if not yet authorized.

## 9. On-Device Model Manager & Hugging Face Authentication

OpenDroid includes an integrated manager for local LiteRT-LM models.

### 🔑 Hugging Face Authentication Setup
To download gated models (such as Google's Gemma 3n or Gemma 4 variants):
1. Generate an **Access Token** (Read permission) on your Hugging Face account settings page.
2. Open OpenDroid's **Settings** screen and scroll down to the **Hugging Face Authentication** card.
3. Paste your token (masked by default, with toggle to show) and click **Validate Token**.
4. The system will query the HF api and display `✓ Token Valid`. The token is stored securely via Android's local KeyStore.

### 📥 Downloading Models
* Tap **Download** on any registered model in the LiteRT-LM models list.
* If a model is gated and you haven't configured a Hugging Face token, the app will display a prompt to guide you to settings.
* Live status checks display download speed (MB/s), downloaded bytes, and remaining time (ETA). Tap **Pause** to suspend or **Cancel** to abort.

### 📋 Offline Local Model Importing
If you already have a model file locally (such as a custom `.task` or `.litertlm` file):
1. Tap **Import** on the model card in Settings.
2. Select your file from the system file picker.
3. The app will copy it to secure sandboxed storage and execute an automated LiteRT load verification check.
4. Once verified, the status changes to `Ready` and it is registered for offline inference.
